# V2 Architecture: Mastra Multi-Agent Research and Document Chatbot

## Goal

Build a production-shaped interview assignment that demonstrates clear system design:

- Answer research questions using external lookup and cited synthesis.
- Analyze uploaded PDFs, PowerPoints, Excel files, CSV files, and Word documents.
- Use Mastra with a multi-agent architecture where agents are used only when reasoning, planning, or synthesis is valuable.
- Use deterministic functions and Mastra tools for parsing, retrieval, filtering, indexing, and data access.
- Support multiple retrieval strategies by document type instead of forcing every file through one generic RAG path.

This design intentionally avoids over-agenting. File ingestion, extraction, indexing, tabular querying, and citation building are deterministic platform capabilities. Agents sit above those capabilities and decide what to do with the evidence.

## Non-Goals

- Full enterprise security, authentication, and authorization.
- Distributed infrastructure as a hard dependency for the assignment.
- Perfect extraction for every edge-case PDF, chart, or Office layout.
- A massive agent graph where every file type gets its own agent without a real reason.

## Architecture Summary

```text
User
  |
  | HTTP /chat, /upload, /documents
  v
API Server
  |
  | enqueue upload job
  v
Ingestion Job Queue
  |
  | extract + normalize + index
  v
Document Processing Pipeline
  |
  | writes artifacts + retrieval indexes
  v
Session Document Store
  |
  | bounded Mastra tools
  v
Mastra Agent Layer
  |
  | Coordinator Agent
  |   -> Research Agent when web research is needed
  |   -> Document Analyst Agent when session documents are needed
  |
  v
Final Answer With Evidence
```

The important design point: **agents do not read raw files**. They call bounded tools over normalized artifacts. This makes answers explainable, limits context bloat, and gives the system a clean testing surface.

## Major Components

| Component | Responsibility | Agent? |
| --- | --- | --- |
| API server | Accept chat, upload, document status requests | No |
| Upload service | Store original file and create file record | No |
| Job queue | Run extraction asynchronously | No |
| Extractors | Parse PDF, DOCX, PPTX, Excel, CSV into normalized artifacts | No |
| Indexers | Build text, metadata, and structured retrieval indexes | No |
| Retrieval tools | Search/query bounded evidence by session and file | No |
| Coordinator Agent | Understand user intent, plan tool use, choose specialist | Yes |
| Research Agent | Web/source lookup, source comparison, research synthesis | Yes |
| Document Analyst Agent | Multi-document reasoning, evidence selection, final grounded answer | Yes |

## Agent Boundary

Agents are used only where natural-language reasoning is useful.

### Coordinator Agent

Responsibilities:

- Classify the request: research, document analysis, hybrid, upload/status, or clarification.
- Decide whether the answer needs web evidence, session document evidence, or both.
- Route to the Research Agent or Document Analyst Agent.
- Merge specialist outputs into a concise final answer.
- Refuse to answer from memory when tools are required.

The Coordinator should not:

- Extract files.
- Scan raw markdown itself.
- Query spreadsheets directly unless using a typed tool.
- Become a giant all-purpose agent with every responsibility.

### Research Agent

Responsibilities:

- Use web research tools for current or external information.
- Prefer credible sources and return cited findings.
- Compare conflicting sources when needed.
- Produce compact research notes for the Coordinator.

Why this is an agent:

- Research questions often need judgment: source quality, recency, conflicting claims, and synthesis.

### Document Analyst Agent

Responsibilities:

- Answer questions from uploaded session documents.
- Choose the correct retrieval path by document type.
- Combine text evidence, spreadsheet facts, slide structure, and citations.
- Call out missing, partial, failed, or unsupported extraction states.
- Produce answers grounded in `EvidencePacket` objects.

Why this is an agent:

- Cross-document reasoning and answer synthesis benefit from LLM reasoning.
- Retrieval itself remains deterministic and tool-driven.

## What Should Not Be an Agent

These should be functions, services, or Mastra tools:

| Capability | Reason |
| --- | --- |
| Upload initialization | Deterministic file bookkeeping |
| File type detection | Rule-based |
| PDF/DOCX/PPTX/Excel extraction | Parser/runtime work, not reasoning |
| Markdown chunking | Deterministic normalization |
| Embedding/index creation | Data pipeline |
| Spreadsheet filters/sorts/grouping | Structured query logic |
| Manifest updates | State management |
| Citation formatting | Deterministic evidence contract |

This keeps the system lean: **few agents, many reliable tools**.

## Async Ingestion and Job Model

Uploads should not block the request thread. The upload API should create a file record, persist the original, enqueue extraction, and return immediately.

```text
POST /upload
  -> validate request
  -> store original file
  -> create manifest/file row
  -> enqueue ExtractDocumentJob
  -> return 202 { fileId, status: "queued" }

Worker: ExtractDocumentJob
  -> mark file "extracting"
  -> detect file kind
  -> run extractor
  -> normalize artifacts
  -> build indexes
  -> mark file "ready" or "partial" or "failed"
```

### Job Statuses

| Status | Meaning |
| --- | --- |
| `queued` | File accepted, extraction not started |
| `extracting` | Parser is running |
| `indexing` | Artifacts exist, retrieval indexes are being built |
| `ready` | File is fully queryable |
| `partial` | Some evidence exists, but extraction has warnings |
| `failed` | No reliable retrieval path is available |
| `unsupported` | File type is intentionally not handled |

### Job Design

For the assignment, an in-process queue is enough:

- `UploadService` pushes jobs into a local queue.
- `DocumentWorker` processes jobs with bounded concurrency.
- Job progress is written into the manifest or metadata store.

For a production evolution, replace the local queue with BullMQ, SQS, Cloud Tasks, or another durable queue. The rest of the system should not care because workers consume the same `ExtractDocumentJob` contract.

### Idempotency

Use content hash as the stable `fileId`:

- Same file in the same session should not duplicate extraction.
- If extraction was already completed for the same hash and parser version, reuse artifacts.
- If parser version changes, re-index or re-extract.

## Document Store and Artifact Model

Each session owns a document workspace:

```text
data/
  users/<userId>/sessions/<sessionId>/
    manifest.json
    uploads/
      <fileId>/original.<ext>
    extracted/
      pdf/<fileId>/
      docx/<fileId>/
      pptx/<fileId>/
      excel/<fileId>/
    indexes/
      text/
      vector/
      tabular/
    workspace/
```

The manifest remains the source of truth for the assignment, but the design should be ready to move metadata into SQLite/libsql:

- File metadata.
- Extraction status.
- Parser version.
- Artifact paths.
- Warnings and errors.
- Index readiness.

Artifacts should be append-friendly and reproducible:

- Original files are immutable.
- Extracted markdown/JSON is derived.
- Indexes can be rebuilt from artifacts.

## Retrieval Strategy by Document Type

The core RAG decision: use the retrieval method that matches the document structure.

### PDF

Artifacts:

- Per-page markdown.
- Per-page structure JSON.
- Optional rendered page images.
- Extraction warnings per page.

Retrieval:

- Keyword/BM25 search over page chunks for precise terms.
- Vector search over semantic chunks for conceptual questions.
- Page-level metadata filters for citations.
- Optional vision extraction for scanned or low-text pages.

Trade-off:

- Text extraction is cheap and fast but weak on scanned/layout-heavy PDFs.
- Vision extraction is expensive but better for forms, tables, and visual layout.
- Page-level chunking gives clean citations but can miss cross-page context.

### DOCX

Artifacts:

- Document markdown.
- Paragraph/block chunks.
- Heading hierarchy where available.

Retrieval:

- Heading-aware chunking.
- Keyword search for exact terms.
- Vector search for semantic questions.

Trade-off:

- DOCX is usually text-rich, so classic text RAG works well.
- Over-small chunks lose section context.
- Over-large chunks waste context window.

### PPTX

Artifacts:

- Slide markdown.
- Slide structure JSON.
- Extracted images/media.
- Best-effort chart summaries.

Retrieval:

- Slide-level search.
- Object-level search for titles, bullets, speaker notes, and chart labels.
- Chart data tool when native chart XML is available.

Trade-off:

- Slide-level chunks give excellent citations.
- XML text extraction misses visual positioning and some embedded content.
- Rendering slides improves fidelity but adds runtime and dependency complexity.

### Excel and CSV

Artifacts:

- Workbook metadata.
- Sheet schemas.
- Row stores.
- Column statistics.
- Preview markdown for quick inspection.

Retrieval:

- Structured querying for filters, sorts, counts, and comparisons.
- Schema inspection before row retrieval.
- Summary/statistical tools for numeric columns.
- Text search only as a fallback, not the primary path.

Trade-off:

- Embedding spreadsheet rows is usually the wrong default.
- Structured tools preserve exact values and reduce hallucination.
- For very large sheets, row stores should move from JSONL to SQLite/DuckDB.

### Hybrid Questions

Some questions need multiple retrieval paths:

Example:

> "Which candidate files are not ready for closure, and what evidence supports that?"

Plan:

- Use spreadsheet schema/query tools for candidate statuses.
- Use document text search for supporting narrative evidence.
- Merge rows and text citations into one `EvidencePacket`.
- Answer with exact row references plus document snippets.

This is where the Document Analyst Agent is valuable: not for raw retrieval, but for deciding how to combine evidence.

## Evidence and Citation Model

Every answer should be traceable to bounded evidence. Tools should return structured evidence, not loose text blobs.

```ts
type EvidencePacket = {
  evidenceId: string;
  source: {
    userId: string;
    sessionId: string;
    fileId: string;
    originalFilename: string;
    documentType: "pdf" | "docx" | "pptx" | "excel" | "csv";
  };
  locator: {
    page?: number;
    slide?: number;
    sheetId?: string;
    rowNumber?: number;
    column?: string;
    blockId?: string;
    chartId?: string;
  };
  content: {
    text?: string;
    row?: Record<string, unknown>;
    table?: Record<string, unknown>[];
    summary?: string;
  };
  score?: number;
  extractionStatus: "ready" | "partial" | "failed";
  warnings?: string[];
};
```

Final answers should cite:

- `fileId`
- original filename
- page, slide, sheet, row, block, or chart locator
- extraction warning if relevant

Example citation format:

```text
[candidate_tracker.xlsx | sheet=candidates | row=42]
[policy.pdf | page=7 | block=3]
[deck.pptx | slide=12 | chart=chart_002]
```

## End-to-End Flows

### Upload Flow

```text
User uploads file
  -> API stores original
  -> Manifest gets queued file record
  -> Job queue receives ExtractDocumentJob
  -> Worker extracts artifacts
  -> Worker builds indexes
  -> Manifest updates to ready/partial/failed
```

### Document Question Flow

```text
User asks document question
  -> Coordinator classifies as document analysis
  -> Coordinator calls Document Analyst Agent
  -> Document Analyst lists session files
  -> Document Analyst chooses retrieval tools by document type
  -> Tools return EvidencePackets
  -> Document Analyst synthesizes grounded answer
  -> Coordinator returns final answer with citations
```

### Research Question Flow

```text
User asks research question
  -> Coordinator classifies as research
  -> Coordinator calls Research Agent
  -> Research Agent searches web
  -> Research Agent compares and summarizes sources
  -> Coordinator returns final answer with source citations
```

### Hybrid Flow

```text
User asks question requiring web + uploaded docs
  -> Coordinator decomposes into research subtask and document subtask
  -> Research Agent returns external context
  -> Document Analyst Agent returns session evidence
  -> Coordinator merges both, clearly separating external sources from uploaded-file evidence
```

## RAG Approaches and Trade-Offs

The project should present multiple RAG approaches, even if the first implementation starts simple.

| Approach | Best For | Weakness |
| --- | --- | --- |
| Keyword/BM25 search | Exact terms, IDs, names, policy clauses | Weak on paraphrases |
| Vector search | Conceptual questions and fuzzy language | Can retrieve plausible but imprecise chunks |
| Hybrid search | Most PDF/DOCX/PPTX questions | More moving parts |
| Structured query | Excel/CSV facts, counts, filters, dates | Requires schema detection |
| Metadata filtering | Page, slide, sheet, file-type scoping | Depends on good extraction metadata |
| Vision extraction | Scanned PDFs and visual layouts | Slower and more expensive |

Recommended assignment implementation:

- Start with keyword search and structured spreadsheet tools.
- Add vector search as a second retrieval path for text-heavy documents.
- Use hybrid ranking for final document retrieval.
- Keep Excel/CSV on structured querying, not embeddings.

Interview explanation:

- "I did not use one universal RAG strategy because document types behave differently. PDFs and Word docs are text retrieval problems, PowerPoints are slide/object retrieval problems, and spreadsheets are structured query problems."

## Observability and Evaluation

For interview polish, add a small evaluation harness:

- Fixed sample questions.
- Expected evidence files or rows.
- Whether the answer included citations.
- Whether extraction status was respected.
- Latency per tool call.
- Retrieval traces showing which tool produced which evidence.

Useful logs:

- Upload accepted.
- Job started/completed/failed.
- Extractor duration.
- Indexing duration.
- Tool call name, input bounds, result count.
- Final answer citations.

This gives a credible answer when asked, "How would you know it works?"

## Practical V2 Build Plan

### Phase 1: Clean Architecture Without Infrastructure Bloat

- Keep local filesystem artifact storage.
- Add in-process async job queue.
- Add `queued` and `indexing` statuses.
- Split ingestion from extraction.
- Keep manifest JSON but make writes atomic.
- Keep current extractors.

### Phase 2: Better Retrieval

- Add persistent text index per session.
- Add vector index for PDF/DOCX/PPTX chunks.
- Keep Excel/CSV structured querying.
- Return `EvidencePacket` from every retrieval tool.

### Phase 3: Cleaner Mastra Layer

- Replace many file-type agents with one `DocumentAnalystAgent`.
- Keep `ResearchAgent` for web work.
- Keep `CoordinatorAgent` for routing and synthesis.
- Move deterministic behavior into tools.

### Phase 4: Interview-Ready Polish

- Add architecture diagram to README.
- Add sample questions and expected retrieval traces.
- Add a `/documents` status view.
- Add clear examples of citations in final answers.
- Document trade-offs in this file and README.

## Final Design Principle

The strongest version of this project is not "many agents talking to each other."

It is:

> A small number of useful agents orchestrating a reliable document-processing and retrieval platform through typed tools.

That is the system design story worth presenting.
